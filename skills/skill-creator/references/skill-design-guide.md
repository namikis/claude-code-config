# Skill設計ガイド

SKILL.md作成時に参照する設計原則とベストプラクティス。

## 1. フロントマター設計

### 必須フィールド

```yaml
---
name: skill-name    # 最大64文字、小文字英数字+ハイフン。/skill-nameで呼び出される
description: ...    # 最大1024文字。自動起動判定の核
---
```

### オプションフィールド（必要な場合のみ）

| フィールド | 値 | 用途 |
|-----------|-----|------|
| `argument-hint` | `[引数の説明]` | 補完時のヒント表示 |
| `disable-model-invocation` | `true` | 手動起動のみに制限。デプロイ・DB変更等の副作用がある操作に使用 |
| `user-invocable` | `false` | `/`メニューに非表示。AI専用の背景知識スキル向け |
| `allowed-tools` | `Bash(npm *), Read` | スキル実行時に許可なく使えるツール |
| `context` | `fork` | サブエージェントで実行。情報収集系・メインコンテキスト汚染を避けたい処理向け |
| `agent` | `Explore`, `Plan`, `general-purpose` | `context: fork`時のエージェントタイプ |
| `model` | モデルID | スキル実行時のモデル指定 |

**注意**: version, author等のメタデータは `metadata:` の下にネストすること。トップレベルに書くとバリデーションエラーになる。

### description最適化（最重要）

Claude Codeはセッション開始時に全スキルのdescriptionを読み込み、ユーザーメッセージとセマンティック（意味的）に照合して自動起動を判定する。キーワード完全一致ではなく意味の近さで判断される。

**全スキルのdescription合計はコンテキスト全体の約2%（最大16,000文字）を常時占める。** スキルが増えるほどこのコストは積み重なるため、情報密度を高めることが重要。

**テンプレート:**
```
{何をするか}。{どんなときに使うか}。「{トリガーフレーズ1}」「{トリガーフレーズ2}」などの指示で使用。{トリガーすべきでない場合: DO NOT TRIGGER when: ...}
```

**良い例:**
```
本番環境へのデプロイを実行するスキル。「デプロイして」「本番に上げて」「リリースして」などの指示で使用。DO NOT TRIGGER when: ローカル環境の構築やテスト実行の場合。
```

**悪い例:**
```
Deploy the application.
```
→ 使用シーン不明、トリガーワード欠如

**5つのポイント:**
1. 体言止めまたは三人称で記述する
2. 複数のトリガーフレーズを含める（日英両方も検討）
3. 使用シーンを明確にする
4. 制約があれば明記する
5. トリガーすべきでない場合を `DO NOT TRIGGER when:` で明記し誤起動を防ぐ

## 2. 本文の設計原則

### Progressive Disclosure（段階的開示）

スキルは「必要になったときに初めてロードされる専門知識パッケージ」として設計する：

1. **description** → セッション開始時に常時ロード（トリガー判定用）
2. **SKILL.md本文** → スキル起動時にロード（実行指示用）
3. **references/** → SKILL.md内で参照指示があった場合にロード（詳細知識用）

この3段階を意識し、各レイヤーに適切な情報を配置することでコンテキスト効率を最大化する。

### 記述ルール

- **500行以内**: 超える場合はreferences/に分離する
- **命令形**: 「〜すること」「〜する」で記述。曖昧な表現を避ける
- **理由の明記**: 単なるルール列挙ではなく、なぜそうするかを簡潔に説明する。Claudeは理由を理解すると応用が効く
- **構造化**: 見出し・リスト・テーブル・コードブロックで視認性を確保する
- **過不足なし**: 必要な指示だけを含める。過剰な注意書きやエッジケースの網羅は避ける
- **明確な完了条件**: 「調査する」ではなく「3件以上のソースを収集し比較表を作成する」のように具体的に

## 3. ディレクトリ構成パターン

### 単純なスキル（SKILL.mdのみ）

```
skill-name/
└── SKILL.md
```

ほとんどのスキルはこれで十分。

### 複合スキル（3層構造）

```
skill-name/
├── SKILL.md          # オーケストレーション手順
├── references/       # ドメイン固有ルール・フォーマット定義・詳細ドキュメント
├── scripts/          # ヘルパースクリプト（繰り返し現れる処理のバンドル）
└── assets/           # テンプレート・設定ファイル等
```

references/を使う場合、SKILL.md内で「〜の詳細はreferences/xxx.mdを参照」と明示的に誘導する。

## 4. 設計パターン集

| パターン | 用途 | ポイント |
|----------|------|----------|
| ワークフロー自動化 | ルーチン手順のバンドル | 明確な成功基準を定義 |
| テンプレート駆動 | 定型出力の生成 | テンプレートはreferences/に配置、SKILL.mdはwhen/whyに集中 |
| パイプライン | 長い処理チェーン | 中間結果をファイルに保存、完了条件を明示 |
| サブエージェント分離 | 情報収集・レビュー等 | `context: fork`で実行、メインコンテキスト汚染を防止 |
| ガードレール | 事前チェック・バリデーション | ローカルCI/CDとして機能、チェックリスト形式 |

## 5. アンチパターン

- **SKILL.md肥大化**: 500行超はコンテキストを圧迫する → references/に分離
- **曖昧なdescription**: 抽象的な説明は自動トリガーを殺す → 具体的な状況とフレーズを記述
- **手動起動依存**: `/skill-name`での起動ばかりになるのはdescription設計の失敗 → description改善
- **過剰な権限**: allowed-toolsを広く設定しすぎる → 必要最小限に制限
- **状態の喪失**: 長時間タスクで中間結果を保存しない → ファイルにチェックポイント保存

## 6. CLAUDE.md / Skills / Subagents の使い分け

| 仕組み | 用途 | ロードタイミング |
|--------|------|-----------------|
| CLAUDE.md | コーディング規約、リポジトリ構造、常時適用ルール | 常時 |
| Skills | タスク単位のワークフロー、手順、判断基準 | オンデマンド |
| Subagents | 独立した試行錯誤、メインコンテキスト汚染回避 | タスク委譲時 |

## 7. セキュリティ

サードパーティ製スキルは信頼性を確認してからインストールする。確認対象：
- SKILL.mdの指示内容
- scripts/の外部通信
- references/へのインジェクションリスク

信頼度: 自作 > チームレビュー済 > Anthropic公式 > 主要OSS > それ以外は要監査
